# 工业4.0 SCADA系统 — 四层漏斗架构重构方案

## 一、当前架构问题分析

### 1.1 单体架构缺陷

```
当前架构（单体）：
┌─────────────────────────────────────────────────────────┐
│                    run.py (主进程)                       │
├─────────────────────────────────────────────────────────┤
│  Modbus轮询 → SQLite写入 → OEE计算 → Flask API → 前端   │
│  S7连接    ↗                                          ↘ │
│  OPC UA   ↗    所有模块在同一进程，I/O阻塞直接影响业务    ↘│
└─────────────────────────────────────────────────────────┘
```

**致命问题：**
1. **协议解析阻塞业务** — Modbus RTU轮询（串口）可能阻塞数百毫秒，直接卡死OEE计算
2. **无故障隔离** — 任何一个模块崩溃（如数据库死锁），整个系统瘫痪
3. **扩展性为零** — 无法水平扩展，无法分布式部署
4. **技术债务累积** — 所有代码耦合，修改一处影响全局

### 1.2 与真实工业现场的差距

| 维度 | 当前系统 | 工业级要求 |
|------|----------|------------|
| 协议支持 | 单进程多协议 | 独立网关进程/硬件 |
| 数据传输 | 直接函数调用 | 消息队列异步解耦 |
| 数据存储 | SQLite | 时序数据库(TDengine/InfluxDB) |
| 设备模型 | 各协议独立解析 | 统一物模型JSON Schema |
| 故障隔离 | 无 | 进程级/容器级隔离 |
| 可扩展性 | 单机单进程 | 微服务/容器编排 |

## 三、重构实施计划

### 阶段一：协议网关服务化（1-2周）

**目标：** 将协议解析从主进程剥离，创建独立的网关服务

**步骤：**
1. 创建 `gateway/` 目录，包含各协议网关
2. 每个网关独立进程运行，通过MQTT发布数据
3. 主系统改为订阅MQTT获取数据

**文件结构：**
```
gateway/
├── __init__.py
├── base_gateway.py          # 网关基类
├── modbus_gateway.py        # Modbus RTU/TCP网关
├── s7_gateway.py            # 西门子S7网关
├── opcua_gateway.py         # OPC UA网关
├── mqtt_gateway.py          # MQTT设备网关
├── thing_model.py           # 统一物模型定义
├── config.yaml              # 网关配置
└── run_gateway.py           # 网关启动脚本
```

### 阶段二：消息总线集成（1周）

**目标：** 引入MQTT Broker，实现发布订阅模式

**步骤：**
1. 安装配置EMQX（开源MQTT Broker）
2. 定义MQTT Topic规范
3. 网关发布数据到MQTT
4. 主系统订阅MQTT获取数据

**MQTT Topic规范：**
```
scada/devices/{device_id}/telemetry    # 设备遥测数据
scada/devices/{device_id}/status       # 设备状态
scada/alarms/{level}                   # 报警信息
scada/oee/{device_id}                  # OEE计算结果
scada/predictive/{device_id}           # 预测性维护结果
scada/energy/{area}                    # 能源数据
```

### 阶段三：时序数据库集成（1-2周）

**目标：** 替换SQLite为专业时序数据库

**步骤：**
1. 安装TDengine/InfluxDB
2. 设计数据模型（超级表/测量值）
3. 迁移历史数据
4. 修改查询逻辑

**TDengine数据模型：**
```sql
-- 超级表：设备遥测数据
CREATE STABLE device_telemetry (
    ts TIMESTAMP,
    value DOUBLE,
    quality INT
) TAGS (
    device_id BINARY(64),
    register_name BINARY(64),
    unit BINARY(16)
);

-- 子表：每个设备的每个寄存器一张表
CREATE TABLE cnc_001_temperature USING device_telemetry TAGS ('CNC_001', 'temperature', '°C');
```

### 阶段四：主系统微服务化（2-3周）

**目标：** 将主系统拆分为独立微服务

**服务拆分：**
```
scada-services/
├── gateway-service/         # 协议网关服务
├── data-service/            # 数据存储服务
├── oee-service/             # OEE计算服务
├── predictive-service/      # 预测性维护服务
├── alarm-service/           # 报警服务
├── api-gateway/             # API网关
└── web-frontend/            # 前端应用
```

## 六、风险与注意事项

1. **渐进式重构** — 不要一次性重写，逐步迁移
2. **向后兼容** — 保持API接口不变，前端无感知
3. **数据迁移** — 历史数据从SQLite迁移到TDengine
4. **测试覆盖** — 每个阶段都要有完整的测试
5. **监控告警** — 新架构需要完善的监控体系
