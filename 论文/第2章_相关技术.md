## 2.1 Modbus通信协议

### 2.1.1 Modbus协议概述

Modbus协议是由Modicon公司（现施耐德电气）于1979年提出的一种串行通信协议，是全球第一个真正用于工业现场的总线协议。经过四十多年的发展，Modbus协议已经成为工业领域通信协议的业界标准，是工业电子设备之间最常用的连接方式之一。

Modbus协议具有以下特点：

（1）**开放性**：协议规范公开发布，无版权要求，任何厂商都可以免费使用。

（2）**简单性**：协议帧格式简单，易于理解和实现，开发成本低。

（3）**可靠性**：支持CRC校验，确保数据传输的可靠性。

（4）**灵活性**：支持多种物理层接口，包括RS-232、RS-485、以太网等。

（5）**广泛性**：几乎所有工业设备都支持Modbus协议，具有极好的兼容性。

### 2.1.2 Modbus通信模型

Modbus协议采用主从（Master-Slave）通信模型。在该模型中，通信过程由主站发起，从站响应。主站发送请求帧，从站接收并处理后返回响应帧。同一时刻只能有一个主站发送请求，从站之间不能直接通信。

主从通信的基本流程如下：

（1）主站向指定从站发送请求帧，包含从站地址、功能码和数据。

（2）从站接收请求帧，检查地址是否匹配。

（3）如果地址匹配，从站执行请求的操作，并返回响应帧。

（4）如果地址不匹配，从站忽略该请求。

（5）主站接收响应帧，检查数据的完整性和正确性。

### 2.1.3 Modbus数据模型

Modbus协议定义了四种基本数据类型，如表2-1所示。

**表2-1 Modbus数据类型**

| 数据类型 | 地址范围 | 访问权限 | 数据宽度 | 说明 |
|---------|---------|---------|---------|------|
| 线圈（Coil） | 00001-09999 | 读/写 | 1位 | 布尔值，用于控制开关量 |
| 离散输入（Discrete Input） | 10001-19999 | 只读 | 1位 | 布尔值，用于采集开关状态 |
| 输入寄存器（Input Register） | 30001-39999 | 只读 | 16位 | 用于采集模拟量 |
| 保持寄存器（Holding Register） | 40001-49999 | 读/写 | 16位 | 用于存储参数和数据 |

### 2.1.4 Modbus功能码

功能码（Function Code）定义了主站请求的操作类型。常用的功能码如表2-2所示。

**表2-2 Modbus常用功能码**

| 功能码 | 操作 | 数据类型 | 说明 |
|-------|------|---------|------|
| 0x01 | 读线圈状态 | 线圈 | 读取1-2000个连续线圈状态 |
| 0x02 | 读离散输入 | 离散输入 | 读取1-2000个连续离散输入 |
| 0x03 | 读保持寄存器 | 保持寄存器 | 读取1-125个连续保持寄存器 |
| 0x04 | 读输入寄存器 | 输入寄存器 | 读取1-125个连续输入寄存器 |
| 0x05 | 写单个线圈 | 线圈 | 写入单个线圈 |
| 0x06 | 写单个寄存器 | 保持寄存器 | 写入单个保持寄存器 |
| 0x0F | 写多个线圈 | 线圈 | 写入多个连续线圈 |
| 0x10 | 写多个寄存器 | 保持寄存器 | 写入多个连续保持寄存器 |

在本系统中，主要使用功能码0x03（读保持寄存器）和0x04（读输入寄存器）来采集现场设备的数据。

### 2.1.5 Modbus RTU帧格式

Modbus RTU是Modbus协议在串行链路上的实现方式，采用二进制编码。RTU帧格式如表2-3所示。

**表2-3 Modbus RTU帧格式**

| 字段 | 长度 | 说明 |
|------|------|------|
| 从站地址 | 1字节 | 从站地址（1-247） |
| 功能码 | 1字节 | 操作类型 |
| 数据 | N字节 | 请求/响应数据 |
| CRC校验 | 2字节 | CRC-16校验码 |

CRC-16校验算法的实现步骤如下：

（1）预置CRC寄存器为0xFFFF。

（2）将第一个数据字节与CRC寄存器低字节进行异或运算。

（3）右移CRC寄存器一位，检查最低位。

（4）如果最低位为1，CRC寄存器与0xA001进行异或运算。

（5）重复步骤（3）和（4），共处理8次。

（6）对下一个数据字节重复步骤（2）至（5）。

（7）处理完所有数据字节后，最终CRC值即为校验码。

### 2.1.6 Modbus TCP帧格式

Modbus TCP是Modbus协议在以太网上的实现方式，使用TCP/IP协议栈进行数据传输。Modbus TCP在RTU帧的基础上增加了MBAP（Modbus Application Protocol）头，去除了CRC校验（由TCP协议保证数据完整性）。

Modbus TCP帧格式如表2-4所示。

**表2-4 Modbus TCP帧格式**

| 字段 | 长度 | 说明 |
|------|------|------|
| 事务标识符 | 2字节 | 请求/响应配对标识 |
| 协议标识符 | 2字节 | 0x0000表示Modbus协议 |
| 长度 | 2字节 | 后续字节数 |
| 单元标识符 | 1字节 | 从站地址 |
| 功能码 | 1字节 | 操作类型 |
| 数据 | N字节 | 请求/响应数据 |

Modbus TCP使用502端口进行通信，支持多连接并发，适用于以太网环境。

## 2.2 SCADA系统架构

### 2.2.1 SCADA系统定义与功能

SCADA（Supervisory Control And Data Acquisition）系统，即数据采集与监视控制系统，是以计算机为基础的生产过程控制与调度自动化系统。SCADA系统的核心功能包括：

（1）**数据采集**：通过与现场设备的通信，实时获取工艺参数和设备状态。

（2）**过程监控**：通过图形化界面直观显示生产过程状态。

（3）**报警管理**：当工艺参数超出正常范围时，及时发出报警信号。

（4）**数据存储**：将采集的历史数据存储到数据库中。

（5）**报表生成**：根据历史数据生成各种统计报表。

（6）**远程控制**：通过网络实现对现场设备的远程控制。

### 2.2.2 SCADA系统层次结构

典型的SCADA系统采用分层架构，从下到上可以分为四个层次：

（1）**现场设备层**：包括各种传感器、执行器、PLC、RTU等现场设备，负责数据的采集和控制命令的执行。

（2）**通信层**：负责现场设备与上位机之间的数据传输，支持多种通信协议（如Modbus、OPC UA、MQTT等）。

（3）**数据处理层**：负责数据的存储、处理和分析，包括数据库管理、报警处理、历史记录等功能。

（4）**人机交互层**：负责与用户的交互，包括图形化界面、报警显示、报表输出等功能。

### 2.2.3 SCADA系统关键技术

SCADA系统涉及的关键技术包括：

（1）**工业通信技术**：包括Modbus、OPC UA、MQTT等工业通信协议。

（2）**数据库技术**：包括关系数据库（如MySQL、PostgreSQL、SQLite）和时序数据库（如InfluxDB）。

（3）**Web技术**：包括HTML、CSS、JavaScript等前端技术，以及Flask、Django等后端框架。

（4）**数据可视化技术**：包括ECharts、D3.js等数据可视化库。

### 2.3.3 WebSocket实时通信

WebSocket是一种在单个TCP连接上进行全双工通信的协议。与HTTP轮询相比，WebSocket具有以下优势：

（1）**实时性强**：服务器可以主动向客户端推送数据，无需客户端反复请求。

（2）**效率高**：WebSocket连接建立后，数据帧头部很小，传输效率高。

（3）**双向通信**：客户端和服务器可以同时发送数据。

本系统使用Flask-SocketIO实现WebSocket通信，用于实时数据推送：

```python
from flask_socketio import SocketIO, emit

socketio = SocketIO(app, cors_allowed_origins="*")

@socketio.on('connect')
def handle_connect():
    emit('response', {'data': 'Connected'})

def push_realtime_data(data):
    socketio.emit('realtime_data', data)
```

## 2.6 MQTT通信协议

### 2.6.1 MQTT协议概述

MQTT（Message Queuing Telemetry Transport）是一种轻量级的消息传输协议，专为物联网（IoT）场景设计。MQTT协议具有以下特点：

（1）**轻量级**：协议头部开销小，最小仅需2字节，适合带宽有限的网络环境。

（2）**发布/订阅模式**：采用发布/订阅模式，解耦了消息的发送者和接收者。

（3）**可靠性**：支持三种QoS（服务质量）级别，确保消息的可靠传输。

（4）**实时性**：支持实时消息推送，延迟低。

（5）**广泛支持**：几乎所有物联网平台和设备都支持MQTT协议。

### 2.6.2 MQTT通信模型

MQTT协议采用发布/订阅（Publish-Subscribe）通信模型，包含以下角色：

（1）**发布者（Publisher）**：发送消息到指定主题。

（2）**订阅者（Subscriber）**：订阅感兴趣的主题，接收相关消息。

（3）**代理（Broker）**：负责消息的路由和转发，管理主题和订阅关系。

MQTT通信的基本流程如下：

（1）客户端连接到MQTT代理。

（2）客户端订阅感兴趣的主题。

（3）发布者向指定主题发送消息。

（4）代理将消息转发给所有订阅该主题的客户端。

（5）订阅者接收并处理消息。

### 2.6.3 MQTT QoS级别

MQTT协议定义了三种QoS级别，如表2-6所示。

**表2-6 MQTT QoS级别**

| QoS级别 | 名称 | 说明 | 适用场景 |
|---------|------|------|---------|
| 0 | 最多一次 | 消息可能丢失，不保证送达 | 对数据丢失不敏感的场景 |
| 1 | 至少一次 | 消息保证送达，但可能重复 | 大多数工业场景 |
| 2 | 恰好一次 | 消息保证送达且不重复 | 对数据一致性要求高的场景 |

本系统使用paho-mqtt库实现MQTT通信，支持QoS 1级别，确保数据的可靠传输。

```python
import paho.mqtt.client as mqtt

def on_message(client, userdata, msg):
    data = json.loads(msg.payload.decode('utf-8'))
    process_data(data)

client = mqtt.Client()
client.on_message = on_message
client.connect("localhost", 1883)
client.subscribe("scada/+/data", qos=1)
client.loop_start()
```

## 2.7 用户认证技术

### 2.7.1 JWT认证机制

JWT（JSON Web Token）是一种开放标准（RFC 7519），用于在各方之间安全地传输信息。JWT由三部分组成：

（1）**Header（头部）**：包含令牌类型和签名算法。

（2）**Payload（负载）**：包含声明（Claims），如用户ID、角色、过期时间等。

（3）**Signature（签名）**：对头部和负载的签名，用于验证令牌的完整性。

JWT认证流程如下：

（1）用户提交用户名和密码进行登录。

（2）服务器验证凭据，生成JWT令牌返回给客户端。

（3）客户端将JWT令牌存储在本地（如localStorage）。

（4）客户端在后续请求中携带JWT令牌（通常放在Authorization头中）。

（5）服务器验证JWT令牌的有效性，处理请求。

### 2.7.3 密码安全

本系统采用bcrypt算法对密码进行哈希处理。bcrypt是一种基于Blowfish密码的密码哈希函数，具有以下特点：

（1）**自适应性**：可以调整计算复杂度，抵御暴力破解攻击。

（2）**加盐**：每次哈希都使用随机盐值，防止彩虹表攻击。

（3）**安全性**：被广泛认为是最安全的密码哈希算法之一。

```python
import bcrypt

# 哈希密码
password = "user_password"
salt = bcrypt.gensalt()
hashed = bcrypt.hashpw(password.encode('utf-8'), salt)

# 验证密码
is_valid = bcrypt.checkpw(password.encode('utf-8'), hashed)
```
