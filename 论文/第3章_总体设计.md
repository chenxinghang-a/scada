# 第3章 系统总体设计

## 3.1 系统需求分析

### 3.1.1 功能需求

根据工业数据采集与监控系统的实际应用场景，本系统的功能需求如下：

（1）**设备管理功能**：支持设备的Web端可视化管理，包括设备的添加、编辑和删除操作，支持设备连接测试和实时状态监控，提供常用设备模板快速配置寄存器。

（2）**设备控制功能**：支持通过Web界面对工业设备进行远程控制，包括Modbus寄存器写入（参数设置）和线圈控制（开关操作），提供常用预设功能降低操作门槛，所有控制操作需二次确认并记录操作日志，支持基于角色的权限控制。

（3）**数据采集功能**：支持多种工业通信协议的统一接入，包括Modbus TCP/RTU（传统PLC和仪表，支持uint16、int16、float32、float64等数据类型解析）、OPC UA（工业4.0标准协议，支持节点订阅和安全认证）、MQTT（物联网消息协议，支持多主题订阅和JSON解析）和REST HTTP（智能网关HTTP接口，支持多种认证方式），通过协议工厂模式实现设备即插即用。

（4）**实时监控功能**：通过Web界面实时显示设备状态和工艺参数，支持数据的自动刷新，支持趋势图表的实时绘制。

（5）**报警管理功能**：支持报警规则的Web端可视化管理（添加、编辑、删除），支持阈值、条件、级别的配置，当工艺参数超出阈值时自动触发报警，支持报警的确认和处理，支持报警统计和查询，支持邮件和声音通知。

（6）**历史数据功能**：支持历史数据的存储和查询，支持按时间范围、设备、参数等条件进行筛选，支持数据的时间间隔聚合（1分钟、5分钟、1小时、1天）。

（7）**数据导出功能**：支持将设备数据和报警记录导出为CSV格式，便于离线分析和报表生成。

（8）**系统管理功能**：支持系统参数的Web端可视化配置（系统设置、采集参数、数据库策略、日志级别），支持系统状态查看、数据库统计、运行时间显示等功能。

### 3.1.2 性能需求

（1）**实时性**：系统应能够实时采集和显示数据，API响应时间不超过100ms。

（2）**可靠性**：系统应具有良好的容错能力，单个设备的故障不应影响其他设备的正常工作，系统稳定性测试成功率应达到99%以上。

（3）**可扩展性**：系统应支持设备的动态添加和删除，方便系统的扩展。

（4）**易用性**：系统应提供友好的Web界面，操作简单直观，支持响应式布局。

### 3.1.3 安全需求

（1）**输入验证**：对所有用户输入进行验证和过滤，防止SQL注入和XSS攻击。

（2）**数据安全**：使用参数化查询防止SQL注入，对输出内容进行HTML转义。

（3）**访问控制**：预留用户认证接口，可扩展用户权限管理功能。

## 3.2 系统架构设计

### 3.2.1 总体架构

本系统采用五层架构设计，从下到上分别为采集层、存储层、报警层、用户层和展示层。各层之间通过定义良好的接口进行通信，实现了高内聚、低耦合的设计目标。

**采集层**负责与工业现场设备的通信，实现多协议设备数据的统一采集。采集层采用协议工厂模式，包含设备管理器、数据采集器和四个协议客户端模块（Modbus客户端、OPC UA客户端、MQTT客户端、REST HTTP客户端）。设备管理器根据设备配置中的protocol字段自动创建对应的协议客户端，实现设备即插即用。数据采集器区分轮询型（Modbus/REST）和推送型（OPC UA/MQTT）两种采集模式，通过线程安全队列实现采集与处理的解耦。

**存储层**负责数据的持久化存储，基于SQLite数据库实现，包含数据库操作模块、数据归档模块和数据导出模块。数据库操作模块提供数据的增删改查接口，数据归档模块实现历史数据的自动聚合压缩，数据导出模块提供CSV/Excel/JSON格式导出。

**报警层**负责报警检测和通知管理，包含报警管理器、报警规则引擎和通知模块。报警规则从YAML配置文件加载，支持运行时Web端热更新，采用状态机设计避免重复报警。

**用户层**负责身份认证和权限管理，基于JWT双令牌机制实现用户认证，支持RBAC四级角色权限控制（管理员、工程师、操作员、观察者），包含登录锁定、操作日志等安全功能。

**展示层**负责人机交互，采用Bootstrap 5和ECharts实现，包含仪表盘、设备管理、报警管理、历史数据和系统配置等功能页面。

### 3.2.2 采集层设计

采集层的核心职责是与工业现场设备进行通信，获取实时数据。采集层采用协议工厂模式和模块化设计，包含以下核心模块：

（1）**设备管理器模块**：管理所有设备的配置和连接。采用协议工厂模式，根据设备配置中的protocol字段自动创建对应的协议客户端（Modbus TCP/RTU、OPC UA、MQTT、REST HTTP），实现设备即插即用。支持从YAML配置文件加载设备配置，支持设备的动态添加和删除，支持模拟模式和真实模式的切换。

（2）**Modbus客户端模块**：封装pymodbus库，提供统一的Modbus通信接口。支持Modbus TCP（以太网）和Modbus RTU（RS485串口）两种通信方式，支持多种数据类型（uint16、int16、float32、float64）的读取和解码。

（3）**OPC UA客户端模块**：封装asyncua库，实现OPC UA协议通信。在独立线程中运行异步事件循环，通过订阅机制实现数据变更实时推送，支持安全认证和加密通信。

（4）**MQTT客户端模块**：封装paho-mqtt库，实现MQTT消息协议通信。支持订阅多个主题，自动解析JSON格式数据，通过回调机制将数据传递给上层，支持QoS可靠性保障。

（5）**REST HTTP客户端模块**：基于requests库实现，对接智能网关和PLC的HTTP接口。支持Bearer Token、Basic Auth、API Key三种认证方式，内置JSON路径解析器，支持可配置的轮询间隔。

（6）**数据采集器模块**：统一调度所有协议设备的数据采集。区分轮询型（Modbus/REST）和推送型（OPC UA/MQTT）两种采集模式。对于Modbus和REST设备，采用定时轮询；对于OPC UA和MQTT设备，采用事件驱动。所有采集数据通过线程安全队列传递给数据处理线程，实现采集与处理的解耦。

### 3.2.3 存储层设计

存储层的核心职责是数据的持久化存储。存储层基于SQLite数据库实现，包含以下四张数据表：

（1）**实时数据表**：存储每个设备每个寄存器的最新值，用于快速查询当前状态。

（2）**历史数据表**：存储所有采集的数据记录，用于历史查询和趋势分析。

（3）**报警记录表**：存储报警事件和处理记录，用于报警查询和统计。

（4）**设备状态表**：存储设备的状态变化记录，用于设备状态追踪。

### 3.2.4 服务层设计

服务层的核心职责是业务逻辑处理。服务层基于Flask框架实现，采用Blueprint进行模块化组织。服务层包含以下核心模块：

（1）**RESTful API模块**：提供标准化的数据接口，包括设备管理、数据查询、报警管理、系统状态等接口。

（2）**WebSocket模块**：基于Flask-SocketIO实现实时数据推送，将采集的实时数据推送到前端页面。

（3）**报警引擎模块**：实现报警的检测和处理，支持多种报警条件（大于、小于、等于等），支持延迟确认机制。

### 3.2.5 展示层设计

展示层的核心职责是人机交互。展示层采用Bootstrap 5和ECharts实现，包含以下功能页面：

（1）**仪表盘页面**：显示系统概览信息，包括设备统计、数据采集统计、报警统计、运行时间等。

（2）**设备管理页面**：提供Web端可视化设备管理界面，支持设备的添加、编辑和删除操作，支持设备连接测试功能，提供常用设备模板（温度传感器、压力传感器、电力仪表、流量计、变频器、PLC等）快速配置寄存器，支持设备状态的实时查看。

（3）**设备控制页面**：提供设备远程控制界面，支持寄存器写入（参数设置）和线圈控制（开关操作），提供常用预设功能（温度设定、启停控制、阀门开关等），支持批量控制操作，所有控制操作需二次确认并记录操作日志。

（4）**报警管理页面**：显示报警记录列表，支持报警的筛选和确认操作。

（5）**历史数据页面**：显示历史数据趋势图，支持按时间范围和设备进行筛选。

（6）**系统配置页面**：显示系统配置信息，支持系统参数的查看和修改。

## 3.4 数据库设计

### 3.4.1 实时数据表设计

实时数据表（realtime_data）用于存储每个设备每个寄存器的最新值，表结构如表3-3所示。

**表3-3 实时数据表结构**

| 字段名 | 数据类型 | 约束 | 说明 |
|-------|---------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键 |
| device_id | TEXT | NOT NULL | 设备ID |
| register_name | TEXT | NOT NULL | 寄存器名称 |
| value | REAL | - | 数据值 |
| unit | TEXT | - | 单位 |
| timestamp | DATETIME | NOT NULL | 采集时间 |
| created_at | DATETIME | DEFAULT CURRENT_TIMESTAMP | 记录创建时间 |

### 3.4.2 历史数据表设计

历史数据表（history_data）用于存储所有采集的数据记录，表结构与实时数据表相同，便于数据的统一处理。

### 3.4.3 报警记录表设计

报警记录表（alarm_records）用于存储报警事件和处理记录，表结构如表3-4所示。

**表3-4 报警记录表结构**

| 字段名 | 数据类型 | 约束 | 说明 |
|-------|---------|------|------|
| id | INTEGER | PRIMARY KEY, AUTOINCREMENT | 主键 |
| alarm_id | TEXT | NOT NULL | 报警规则ID |
| device_id | TEXT | NOT NULL | 设备ID |
| register_name | TEXT | NOT NULL | 寄存器名称 |
| alarm_level | TEXT | NOT NULL | 报警级别 |
| alarm_message | TEXT | - | 报警消息 |
| threshold | REAL | - | 阈值 |

| timestamp | DATETIME | NOT NULL | 报警时间 |
| acknowledged | BOOLEAN | DEFAULT 0 | 是否已确认 |

| acknowledged_by | TEXT | - | 确认人 |

### 3.4.4 索引设计

为提高查询效率，在以下字段上建立索引：

（1）实时数据表：device_id + timestamp

（2）历史数据表：device_id + timestamp

（3）报警记录表：device_id + timestamp

### 3.5.3 错误处理机制

本系统的错误处理机制包括：

（1）**通信超时处理**：当通信超时时，记录错误日志并跳过本次采集。

（2）**数据校验**：对接收到的数据进行范围检查，过滤异常值。

（3）**设备故障处理**：当设备连续多次采集失败时，标记设备为离线状态。

（4）**自动重连**：当设备恢复后，自动重新建立连接。
